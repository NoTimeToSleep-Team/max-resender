# Raspberry Pi Access

Этот документ описывает, как безопасно выдать доступ к Raspberry Pi для финального деплоя и диагностики.

## Что нужно подготовить

- IP-адрес Raspberry Pi в локальной сети или внешний адрес через VPN/Tailscale;
- SSH-пользователь без sudo по умолчанию или отдельный deploy-пользователь;
- способ повышения прав только через `sudo`;
- подтверждение, что на устройстве уже есть `node`, `npm`, `nginx` и `systemd`.

## Предпочтительный способ доступа

Лучший вариант — отдельный временный пользователь, например `projectmax`, с доступом только к нужной папке и нужным сервисам.

## Минимально нужные права

Нужны права на:

- чтение и запись в папку проекта, например `/opt/project-max`;
- управление backend service через `systemctl restart/status project-max-backend`;
- чтение логов `journalctl -u project-max-backend`;
- чтение и обновление nginx config, если reverse proxy настраивается на месте;
- чтение `.env`, но лучше передавать секреты вручную отдельно, а не широким доступом.

## Как выдать безопасный SSH доступ

1. Создай отдельного пользователя:

```bash
sudo adduser projectmax
```

2. Добавь мой публичный SSH key в:

```bash
/home/projectmax/.ssh/authorized_keys
```

3. Дай права на проект:

```bash
sudo mkdir -p /opt/project-max
sudo chown -R projectmax:projectmax /opt/project-max
```

4. Если нужен `sudo`, лучше ограничить его конкретными командами через `/etc/sudoers.d/projectmax`, а не давать полный root.

Пример ограниченного sudo:

```bash
projectmax ALL=(ALL) NOPASSWD: /bin/systemctl status project-max-backend, /bin/systemctl restart project-max-backend, /bin/journalctl -u project-max-backend
```

5. Если используется nginx, можно отдельно разрешить только чтение/замену конкретного файла конфига, а не полный root-shell.

## Что нужно передать мне для работы

- SSH host или IP;
- username;
- порт, если не `22`;
- способ аутентификации: ключ уже установлен или нужен временный пароль;
- какие команды `sudo` разрешены;
- путь проекта на Raspberry Pi;
- путь к `.env`, если он отличается от `/opt/project-max/.env`.

## Что не стоит делать

- не давать постоянный root-доступ без необходимости;
- не пересылать production секреты в открытом виде, если можно вставить их локально на устройстве;
- не использовать один общий пользователь для всего домашнего сервера и всех сервисов.

## После завершения работ

- удалить временный доступ или SSH key;
- пересмотреть sudo rules;
- при необходимости сменить чувствительные секреты.
